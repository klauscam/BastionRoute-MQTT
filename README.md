# BastionRoute MQTT Shim (v0.1.0-alpha)

This document provides a deep architectural breakdown, design justification, and operational assessment of the `bastionroute-mqtt` system component. 

---

## 1. Architectural Architecture & Design Decisions

The `bastionroute-mqtt` component acts as an outbound-only user-space transport engine. It wraps Layer 3 IP traffic generated by standard virtual network interfaces (like WireGuard tunnels) and marshals those byte streams securely over stateless pub/sub infrastructures.


### Design Decision: User-Space UDP Interception
Rather than writing a custom kernel module or virtual network driver, BastionRoute intercepts traffic via a stateless user-space loop. It binds to a local UDP loopback interface (`127.0.0.1:51820`). 
* **Justification:** This drastically lowers system complexity, eliminates the need for root execution configurations in many non-privileged deployment spaces (like Termux on Android), and ensures universal portability across cross-compilation environments.

### Design Decision: Choosing a Stateless Pub/Sub Fabric over Direct TCP
Traditional proxy fabrics require establishing an addressable server. By substituting a direct TCP connection stream for a pub/sub topology (MQTT), network nodes no longer connect directly to each other. Instead, they connect to independent message pipelines hosted on intermediate brokers.
* **Justification:** This achieves absolute topology decoupling. Endpoints are completely blind to each other’s public IP addresses, geolocation data, and network carriers.

## 2. Cryptographic Security Features & Threat Mitigation

BastionRoute MQTT operates on a strict **Zero-Trust Transport Model**. The underlying message broker is assumed to be actively hostile or compromised by third parties. Security is enforced mathematically via decoupled cryptographic layers.

### Temporal Topic Blinding
Multi-tenant pub/sub environments leak infrastructure structure if topic trees are human-readable. BastionRoute mitigates this via dynamic, time-sliced topic calculations. 

Communication routes are flattened into a single string derived from a Keyed-Hash Message Authentication Code (HMAC) using a SHA-256 underlying hashing architecture. The message space rotates sequentially every 300 seconds (5 minutes) based on a integer timestamp block:

$$T_{\text{block}} = \lfloor \frac{\text{Unix Epoch Timestamp}}{300} \rfloor$$

$$\text{Active Topic Hash} = \text{HMAC-SHA256}(\text{Master Secret}, \text{Room Name} \parallel T_{\text{block}})$$

Because the hashing process is non-reversible, a broker or external analyst cannot associate sequential historical topic streams to map node relationships.

### Absolute Connection Signatures Erasure
To avoid leaving operational footprints inside the broker's active client connection tables, the core engine strips out identifiable application prefixes. 
* Connection Client IDs are computed via a cryptographically secure random sequence generator (`crypto/rand`), yielding a flat, anonymous 16-character hexadecimal string block. 
* To a monitoring system, BastionRoute connections match the formatting configuration of unconfigured generic consumer IoT appliances, preventing signature analysis finger-printing.

### Control and Data Plane Isolation
* **Outer Encapsulation (Control Plane):** Tunnel synchronization frameworks and heartbeats are encrypted with authenticated symmetric block ciphers (AES-GCM). Replay attacks are mitigated by validating sliding nonce tracking registers and verifying epoch timestamps against a strict microsecond clock-skew allowance.
* **Inner Core (Data Plane):** The application payload data is passed transparently as raw bytes directly into the encrypted WireGuard encapsulation matrix (Curve25519, ChaCha20-Poly1305). If the outer wrapper or the broker's transport layer TLS is entirely compromised, data visibility drops back to an unbreachable cryptographic wall.

## 3. Concurrency Model & Flow Control

The shim architecture is built entirely upon Go's native, highly efficient runtime scheduler. The engine relies heavily on lightweight channel allocations and thread-safe memory primitives to scale throughput over high-concurrency loops.


### Thread-Safe Routing & Mapping
The server-side component handles concurrent data streams from multiple incoming client endpoints over a single shared connection layer. This tracking requires low-latency, race-free lookups.
* The system utilizes Go's optimized `sync.Map` storage structures to register active remote client addresses.
* Client tracking keys are derived from the unique incoming cryptographically verified peer identities, safely mapping returned payloads without introducing mutex-locking bottlenecks across concurrent processing pipelines.

### Flow Control & The "TCP Meltdown" Solution
When a protocol requiring packet acknowledgments (like TCP or HTTP) is tunneled inside another stateful protocol loop (like MQTT over TCP), congestion can trigger a failure state known as **TCP Meltdown**. 

If the public MQTT broker experiences structural network latency, the outer TCP layer pauses and retransmits. Meanwhile, the nested HTTP or TCP app detects a delayed packet acknowledgment, assumes network loss, drops its congestion window, and issues a duplicate retry packet. This creates a cascading bottleneck that stalls data streams.

BastionRoute fixes this behavior through its user-space queue design:
1. Data ingested from the kernel interface is dropped into buffered Go channels (`chan []byte`) acting as a non-blocking ring buffer.
2. If the MQTT publisher pipeline detects a bottleneck or high broker latency, the engine deliberately drops old packets from the buffer instead of blocking the execution routine.
3. By introducing intentional packet loss during network degradation, the shim allows the underlying WireGuard client to naturally drop its window limits and activate its native error correction routines. This prevents system freezes and preserves operational throughput.
